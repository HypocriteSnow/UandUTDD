# Unity 关卡编辑器测试边界设计

## 1. 设计目标

本设计用于明确 Unity 版关卡编辑器中“编辑器预览 / 校验 / 测试启动 / 正式 Runtime”之间的职责边界，避免沿用网页端为了弥补运行环境缺失而形成的“试玩大厅 + 编辑器内运行器”模式。

核心目标如下：

1. 编辑器负责生产关卡数据、执行必要校验，并提供便捷的测试启动入口。
2. Unity `Play` 后的真实游玩必须由正式 Runtime 接管，避免维护第二套战斗真相。
3. 测试入口要足够便捷，能够快速选择当前关卡或最近关卡进入测试。

## 2. 背景判断

网页端 `LevelEditor.jsx` 中之所以存在 `PLAY_MENU` 和 `PlayMode`，本质原因是网页环境缺乏项目内正式 Runtime，因此编辑器不得不同时承担：

1. 关卡数据编辑
2. 关卡选择入口
3. 轻量战斗模拟

这套形态在网页端是合理的，但在 Unity 中继续照搬会引入三个长期问题：

1. 编辑器内规则与正式 Runtime 规则逐步分叉。
2. 测试关卡、当前编辑关卡、正式加载关卡之间出现多套状态。
3. 编辑器壳层不断膨胀，重新演化为单体 God Tool。

## 3. 目标方案

Unity 版编辑器应将网页端“试玩大厅”重新拆成两层：

1. **测试启动器**
   - 负责选择关卡、选择测试参数、触发 `Play`。
2. **正式 Runtime 接管**
   - 在 Play Mode 启动后读取测试目标并进入真实战斗流程。

同时，编辑器内部保留一层更轻的**预览与校验**能力：

1. 地图、路径、传送门、波次的可视化反馈
2. 启动前必要校验
3. 必要时的轻量规则探针

这里的重点不是在编辑器里重做完整战斗，而是缩短“改数据 -> 校验 -> Play 测试”的回路。

## 4. 职责边界

### 4.1 编辑器

编辑器负责：

1. 地图、波次、路径、模板、JSON 的编辑
2. 关卡保存、导入、导出
3. 测试目标选择
4. 启动前校验

编辑器不负责：

1. 长期维护完整战斗模拟
2. 成为正式运行时的替身

### 4.2 预览与校验层

预览与校验层负责：

1. 路径和传送门可视化
2. 起点、终点、波次、路径、绑定关系检查
3. 启动测试前的阻断性错误提示

它的输出应服务于“尽快发现问题”，而不是“给出最终权威结果”。

### 4.3 测试启动器

测试启动器负责：

1. 选择当前要测试的关卡
2. 选择最近关卡、已保存关卡或临时导入关卡
3. 写入测试会话数据
4. 触发 Unity Play Mode

### 4.4 Runtime

Runtime 负责：

1. 读取测试会话
2. 加载目标关卡
3. 执行正式 Grid / Nav / Entity / Combat 流程
4. 给出唯一可信的游玩结果

## 5. 状态机建议

Unity 目标版建议不再保留网页端的 `PLAYING` 作为编辑器内状态，而改为：

1. `EDITOR`
2. `JSON`
3. `TEST_LAUNCHER`
4. `PLAY_MODE_RUNTIME`

其中前 3 个属于编辑器 UI，最后 1 个属于点击 `Play` 后的运行时状态。

## 6. 工作流建议

推荐工作流如下：

1. 在编辑器内修改关卡
2. 执行预览与校验
3. 保存当前关卡或生成临时测试快照
4. 在测试启动器中选择目标
5. 点击 `Play`
6. Runtime 读取测试会话并进入正式战斗
7. 退出 Play 后返回编辑器继续修改

## 7. 对主规格的影响

该设计会直接影响主规格中的以下定义：

1. `PLAY_MENU` 改名并转义为 `TEST_LAUNCHER`
2. `PLAYING` 从编辑器内部模式改为 Runtime 接管状态
3. “编辑器内试玩”表述统一改为“预览校验 + 测试启动”
4. 架构层新增 Runtime 接管层
5. 能力清单从“内置试玩”改为“测试启动器 + Play Mode 测试入口”

## 8. 结论

Unity 版关卡编辑器更合理的定位不是“自带完整试玩大厅”，而是：

1. 做好关卡生产
2. 做好预览校验
3. 做好测试启动
4. 把真实游玩交给正式 Runtime

这能显著降低编辑器与游戏运行逻辑分叉的风险，也更符合后续系统化扩展方向。
