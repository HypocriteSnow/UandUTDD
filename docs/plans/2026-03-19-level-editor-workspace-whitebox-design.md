# Unity 关卡编辑器工作区/白模架构设计

## 1. 背景

当前 Unity 关卡编辑器仍然沿用“先准备 `LevelConfig`，再由编辑器读取它并生成可编辑内容”的思路。这种模式适合早期基于配置文件的小型工具，但不适合当前目标中的完整关卡生产流程。

网页编辑器的核心体验不是“编辑某个配置资产”，而是“直接编辑当前关卡本身”。用户先在编辑器中定义关卡尺寸与全局参数，随后立即获得一个可持续编辑的地图工作面，再继续进行笔刷、传送门、路径、波次、模板等精细化操作。

本次设计的目标，是把 Unity 关卡编辑器重构为与这一思路一致的工作流。

## 2. 目标

### 2.1 核心目标

1. 编辑器自身成为关卡创作主入口，而不是 `LevelConfig` 资产的附属查看器。
2. 当用户确定地图尺寸后，编辑器即可立即生成当前关卡的白模工作面。
3. 后续地图笔刷、传送门、路径、波次、敌人模板、干员模板等编辑行为，均围绕白模与编辑器工作区进行。
4. 最终由编辑器将当前工作区导出为运行时配置资产，命名规则为：`当前关卡名称_LevelConfig`。

### 2.2 非目标

1. 不继续强化“每次手动拖入 `LevelConfig` 和 `GridVisualConfig` 才能开始编辑”的旧流程。
2. 不把白模场景本身作为唯一真源数据。
3. 不把所有编辑器状态继续塞回单一 `LevelConfig` 中。

## 3. 核心问题

当前模式存在三个根本问题：

1. **入口错位**
   用户面对的是 `LevelConfig` 资产，而不是“当前正在制作的关卡”。

2. **职责混杂**
   `LevelConfig` 同时承担了地图参数、部分关卡数据、编辑入口、运行时数据源等多种职责。

3. **工作流滞后**
   网页编辑器是“参数变化后立即形成可编辑关卡画布”，而当前 Unity 编辑器仍然是“先读配置，再进入编辑”。

## 4. 推荐架构

推荐将系统拆成三层，并明确主从关系：

1. `LevelEditorWorkspace`
2. `Whitebox Scene / Whitebox Root`
3. `LevelConfig` 导出资产

### 4.1 `LevelEditorWorkspace`

`LevelEditorWorkspace` 是编辑器期的真实数据源。它表示“当前关卡正在被如何编辑”，而不是“运行时最终如何消费”。

它至少应包含：

1. 关卡基础信息
   - `levelName`
   - `displayName`
2. 地图基础参数
   - `mapWidth`
   - `mapDepth`
   - `cellSize`
   - `defaultTileType`
3. 关卡全局参数
   - `initialDp`
   - `baseHealth`
   - `dpRecoveryInterval`
   - `dpRecoveryAmount`
4. 编辑中的地图语义数据
   - 特殊格子
   - 红门/蓝门/传送门
   - 路径节点
   - 波次配置
   - 敌人模板
   - 干员模板

设计原则：

1. 用户在编辑器中修改的首先是 `Workspace`
2. `Workspace` 驱动白模生成和更新
3. 保存/导出时才将其转换为 `LevelConfig`

### 4.2 Whitebox

Whitebox 是编辑期的空间工作面。它可以是独立场景，也可以是场景中的统一根节点，但其职责必须明确为“承载编辑交互”。

当用户确认地图尺寸后，编辑器应立即能够：

1. 创建或刷新白模网格
2. 按照尺寸生成格子表现
3. 在白模上标记起点、终点、传送门等语义对象
4. 允许笔刷、选择、路径绘制等交互直接作用于白模

Whitebox 不是运行时最终配置，也不是唯一真源。它是 `Workspace` 的可视化与操作载体。

### 4.3 `LevelConfig`

`LevelConfig` 的职责应被收缩为“运行时可消费的导出结果”。

它应由编辑器根据当前 `Workspace` 生成，而不是反过来作为编辑器的前置输入。

最终导出命名规则：

`<当前关卡名称>_LevelConfig`

例如：

`Tutorial_01_LevelConfig`

建议导出路径统一到项目配置目录，例如：

`Assets/Resources/Levels/Configs/<当前关卡名称>_LevelConfig.asset`

## 5. 新的编辑工作流

推荐的标准流程如下：

1. 打开关卡编辑器
2. 新建关卡或选择已有关卡工作区
3. 在编辑器中直接设置关卡基础参数
   - 地图长宽
   - 初始 DP
   - 蓝门 HP
   - 回费间隔
   - 单次回费量
4. 当尺寸确认后，立即生成或刷新白模
5. 在白模上进行地图笔刷与精细编辑
6. 继续编辑传送门、路径、波次、敌人模板、干员模板
7. 保存时更新工作区
8. 导出时生成 `<当前关卡名称>_LevelConfig`
9. Unity 运行时或测试入口仅读取导出的 `LevelConfig`

## 6. 关键模块

### 6.1 关卡工作区管理

负责：

1. 新建工作区
2. 加载已有工作区
3. 保存工作区
4. 管理当前关卡名称与导出命名
5. 管理最近一次编辑的关卡

### 6.2 白模生成服务

负责：

1. 根据 `Workspace` 基础参数创建白模
2. 在尺寸变化时刷新白模
3. 在白模与工作区之间建立双向同步接口

### 6.3 地图交互控制器

负责：

1. 笔刷涂抹
2. 单格属性修改
3. 起点/终点/传送门布置
4. 路径节点绘制

### 6.4 导出服务

负责：

1. 将当前 `Workspace` 整理为运行时结构
2. 生成或覆盖 `<关卡名>_LevelConfig`
3. 保证导出结果对 Runtime 清晰、稳定、可校验

### 6.5 校验与测试入口

负责：

1. 校验工作区的结构完整性
2. 校验导出配置的有效性
3. 支持一键导出并进入 Play 测试当前关卡

## 7. 数据边界建议

### 7.1 应属于 `Workspace` 的数据

1. 编辑器正在调整的关卡参数
2. 当前未导出的草稿状态
3. 白模编辑中的中间状态
4. 路径绘制和局部操作上下文

### 7.2 应属于 Whitebox 的数据

1. 编辑期格子对象
2. 可视化辅助对象
3. 传送门标记物
4. 路径节点锚点

### 7.3 应属于 `LevelConfig` 的数据

1. Runtime 必须消费的最终关卡定义
2. 地图尺寸与格子语义
3. 关卡全局参数
4. 波次、路径、传送门、模板等最终配置

### 7.4 不应属于 `LevelConfig` 的数据

1. 当前选中的编辑器 Tab
2. 当前画笔模式
3. 最近一次鼠标交互
4. Whitebox 临时对象状态
5. 仅用于编辑中的草稿上下文

## 8. 与旧架构的关系

当前已经实现的部分内容仍然有复用价值，但需要重新归位：

1. `LevelEditorWindow` 继续保留为壳层入口，但职责需进一步收缩
2. 已有的地图笔刷逻辑需要从“直接写 `LevelConfig`”改为“操作 `Workspace` 并刷新白模”
3. 已有 `LevelValidationService` 可以保留，但校验对象应逐步从 `LevelConfig` 扩展到 `Workspace` 和导出结果
4. 已有 `LevelSceneBuilder` / `LevelSceneLoader` 需要转型为白模生成/同步服务，而不是单纯的“从配置构建场景”

## 9. 测试策略

测试应围绕新数据流重新组织：

1. `Workspace` 创建默认值测试
2. 尺寸修改后白模生成测试
3. 白模笔刷写回工作区测试
4. `Workspace -> LevelConfig` 导出测试
5. 导出命名规则测试
6. 工作区校验与导出校验测试

## 10. 结论

后续 Unity 关卡编辑器的正确方向不应再是“继续扩张 `LevelConfig` 作为编辑中心”，而应当是：

1. 以编辑器工作区作为编辑期真源
2. 以白模作为即时生成、可精修的工作面
3. 以 `LevelConfig` 作为最终运行时导出资产

最终统一工作流可以概括为：

`编辑器参数 -> 白模生成与编辑 -> 导出 <当前关卡名称>_LevelConfig`
